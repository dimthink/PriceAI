# PriceAI 后台管理左侧导航与工作流拆分规划

> 日期：2026-07-12  
> 状态：规划完成，待分阶段实施  
> 文档类型：后台管理 PRD Lite / 信息架构 / UI 结构 / 组件拆分规划  
> 上游规划：[工程质量与可维护性规划](2026-07-10_priceai-engineering-quality-and-maintainability-plan.md)  
> 关联热区：`src/components/AdminConsole.tsx`、`src/components/ApiTransitAdminConsole.tsx`  
> 本轮边界：只沉淀产品与执行规划，不修改后台代码、不拆文件、不变更接口或数据结构

## 1. 产品一句话

这是一个面向 PriceAI 运营者的后台管理工作台改版，帮助运营者在审核、数据维护、API 中转、采集系统和安全配置等高频后台任务中，从横向拥挤的 Tab 操作转为左侧一级/二级分区导航 + 右侧工作区，从而降低认知负担、减少误操作，并为 `AdminConsole.tsx` 的按工作流拆分建立清晰边界。

## 2. 背景判断

当前后台已经从早期的单一审核台演化成完整运营控制台，承载了渠道源、标准商品、报价、官方价、API 模型、中转 API、反馈、采集、安全、历史、赞助等多个工作流。

截图暴露的问题不是单个页面样式不够精致，而是信息架构已经超过横向 Tab 的承载范围：

- 顶部横向 Tab 数量过多，标签、图标和计数挤在同一行，后续再加模块会继续压缩可读性。
- 同一工作区里同时出现一级 Tab、二级胶囊按钮、搜索、状态筛选、统计卡和表格，页面上方变厚，真正的数据列表被推低。
- `API 中转` 这类模块已经有站点池、清洗候选、原始报价、提交线索、检测记录等二级对象，不适合继续作为一个横向 Tab 内的局部筛选。
- `AdminConsole.tsx` 当前约 1 万行，持有太多状态、请求、副作用和 JSX。仅做视觉调整不会解决长期维护问题。

因此，这次后台改版应该把“左侧导航的信息架构改造”和“按工作流拆组件”作为同一个方向推进，但实施时保持小步迁移。

## 3. 产品边界

### 本版本做什么

1. 把后台从顶部横向 Tab 改造成左侧一级/二级导航 + 右侧工作区的 SaaS 后台结构。
2. 按运营工作流重新组织后台模块，让审核、数据维护、API 中转、采集系统和系统设置各有稳定入口。
3. 定义 `AdminConsole` 的目标职责：登录壳、导航、全局状态和跨模块协调，不再直接承载所有表单、请求和业务规则。
4. 规划按模块迁移顺序，避免一次性重写后台造成大面积回归。
5. 为后续 UI handoff、Trellis 实现任务和验收标准提供统一产品上下文。

### 本版本不做什么

1. 不直接修改生产代码。
2. 不重新设计数据库表、API 路由或权限系统。
3. 不把所有后台页面一次性重写。
4. 不把视觉改版、状态迁移、请求层抽象和业务逻辑修复混在同一阶段。
5. 不引入新的复杂后台产品形态，例如独立桌面端、移动 App 或多角色企业权限矩阵。

## 4. 场景问题 -> 用户任务 -> 用户旅程 -> 功能模块 -> 页面结构 -> 具体功能点

### 4.1 场景问题

| 场景 | 当前问题 | 后果 |
| --- | --- | --- |
| 日常审核 | 审核、反馈、待办、赞助等入口并列在横向 Tab 中 | 运营者需要先记住入口位置，再进入具体队列 |
| API 中转维护 | 一个 Tab 内继续塞站点池、候选、原始报价、提交线索、检测记录 | 二级对象不清晰，搜索和筛选语义混杂 |
| 采集排障 | 采集待办、历史、健康状态分散在顶部和内容区 | 采集问题需要跨区域查找上下文 |
| 数据维护 | 渠道源、标准商品、报价、官方价、API 模型都偏维护型，但入口平铺 | 新增能力时容易继续堆到同一个大组件里 |
| 后台扩展 | `AdminConsole.tsx` 同时承担导航、数据请求、状态、表单和具体页面 | 后续任何后台需求都容易触碰 1 万行文件 |

### 4.2 用户任务

| 用户 | 高频任务 | 需要的界面能力 |
| --- | --- | --- |
| PriceAI 运营者 | 查看待审核项、批准、拒绝、导入、标记待办 | 清晰队列、批量处理、详情预览、状态反馈 |
| 数据维护者 | 维护渠道源、标准商品、报价和官方价格 | 稳定表格、搜索筛选、编辑入口、变更确认 |
| API 中转维护者 | 管理站点池、清洗候选、原始报价和检测记录 | 二级导航、分对象列表、快速查看站点详情 |
| 采集维护者 | 查看采集待办、运行历史和健康状态 | 时间序列、失败原因、重试/排障入口 |
| 站点管理员 | 管理安全、历史、公告、赞助配置 | 低频但可靠的系统设置入口 |

### 4.3 核心用户旅程

```text
进入后台
-> 通过左侧一级分区判断任务类型
-> 选择二级对象或队列
-> 在右侧工作区查看指标、筛选、列表
-> 打开详情区或抽屉处理单条记录
-> 保存 / 批量操作 / 标记状态
-> 留在当前分区继续处理下一条
```

| 步骤 | 用户动作 | 系统动作 | 用户获得什么 | 关键风险 |
| --- | --- | --- | --- | --- |
| 1 | 登录后台 | 展示顶部状态和左侧导航 | 明确自己在后台哪个工作区 | 首屏信息过多 |
| 2 | 选择一级分区 | 展开对应二级导航 | 快速定位任务类型 | 分组命名不符合运营心智 |
| 3 | 进入二级页面 | 加载该工作流指标、筛选和列表 | 看到当前待处理对象 | 旧 Tab 状态迁移丢失 |
| 4 | 搜索或筛选 | 更新列表和计数 | 缩小处理范围 | 筛选状态与导航状态冲突 |
| 5 | 查看详情或编辑 | 打开详情区、抽屉或独立表单区 | 不离开队列也能处理单项 | 详情区太重造成页面拥挤 |
| 6 | 保存或批量处理 | 显示明确成功/失败反馈 | 可继续处理下一项 | 错误状态不明确 |

## 5. 目标信息架构

建议采用左侧常驻导航，一级分区保持 5-6 个，二级分区按对象类型组织。

```text
后台管理
├─ 总览
├─ 审核队列
│  ├─ 渠道提交
│  ├─ 报价反馈
│  ├─ 站点反馈
│  └─ 赞助审核
├─ 数据维护
│  ├─ 渠道源
│  ├─ 标准商品
│  ├─ 报价维护
│  └─ 官方价格
├─ API 中转
│  ├─ 站点池
│  ├─ 清洗候选
│  ├─ 原始报价
│  ├─ 提交线索
│  └─ 检测记录
├─ 采集系统
│  ├─ 采集待办
│  ├─ 运行历史
│  └─ 健康状态
└─ 系统
   ├─ 安全
   ├─ 操作历史
   └─ 站点公告
```

### 分组原则

1. 一级分区按运营任务类型命名，而不是按代码模块命名。
2. 二级分区按可操作对象命名，例如站点池、报价反馈、运行历史。
3. 计数只显示在真正有队列意义的入口上，例如待审核、反馈、采集待办。
4. 低频设置进入“系统”，不要继续挤在主业务导航里。
5. API 中转是完整业务域，不再作为一个普通横向 Tab。

## 6. 页面与界面结构

### 6.1 后台整体布局

| 区域 | 主要用途 | 包含内容 | 状态 |
| --- | --- | --- | --- |
| 顶部栏 | 品牌、面包屑、全局计数、登录状态 | 返回前台、后台标题、关键全局指标、解锁/退出 | 正常、会话失效 |
| 左侧导航 | 一级/二级分区入口 | 分组标题、图标、二级项、队列计数、折叠状态 | 展开、折叠、移动端抽屉 |
| 工作区标题 | 当前页面上下文 | 页面标题、说明、关键操作、最近更新时间 | 加载、正常、错误 |
| 指标带 | 当前工作流关键计数 | 待处理、已发布、失败、异常、候选数 | 可选；不强制每页都有 |
| 筛选工具栏 | 搜索与状态筛选 | 搜索框、状态筛选、对象切换、刷新 | 空、筛选中、无结果 |
| 主列表/表格 | 核心数据处理 | 表头、行、行操作、批量操作 | loading skeleton、空、错误、正常 |
| 详情区 | 单条对象处理 | 详情、编辑、审核、历史、风险提示 | 抽屉、右侧详情栏或独立页 |

### 6.2 典型页面节奏

```text
左侧导航
右侧工作区：
  页面标题 + 关键动作
  当前模块指标
  搜索 / 筛选 / 状态切换
  表格 / 队列
  详情抽屉 / 内联详情
```

对截图中的 `API 中转` 工作区，目标不是继续在内容区顶部塞五个胶囊按钮，而是让这些对象成为左侧二级导航：

- `站点池`：展示已发布、待发布、已移除站点。
- `清洗候选`：展示候选站点与清洗状态。
- `原始报价`：展示原始报价、模型、分组、URL。
- `提交线索`：展示用户或站长提交的中转站线索。
- `检测记录`：展示模型检测、可用性、最近运行结果。

## 9. 组件拆分目标

### 9.1 目标职责

`AdminConsole.tsx` 最终只保留：

1. 登录态和后台解锁壳。
2. 顶部栏和左侧导航的当前选中状态。
3. 全局 summary / count 的读取和刷新。
4. 全局 toast/message。
5. 页面级错误边界。
6. 当前工作区组件的挂载。

`AdminConsole.tsx` 不再直接持有：

1. 每个工作流的表单草稿。
2. 每个 endpoint 的 fetch/mutation 细节。
3. 具体表格行的业务规则。
4. 审核、采集、API 中转、赞助等页面的大段 JSX。
5. 所有后台 API 的错误处理分支。

### 9.2 建议目录形态

```text

├─ AdminShell.tsx
├─ AdminSidebar.tsx
```

## 10. 分阶段执行规划

### Phase 0：后台功能地图与迁移边界

目标：在动代码前确认每个旧 Tab、状态、请求和操作的归属。

产出：

1. 旧入口到新页面的映射表。
2. `AdminConsole.tsx` 内部 state / effect / handler / memo 的归属清单。
3. 首个迁移切片的验收路径。

完成标准：

- 每个现有后台功能都有目标页面。
- 没有“暂时不知道放哪”的后台入口。
- 明确哪些功能只是视觉搬迁，哪些涉及状态迁移。

### Phase 1：建立 Admin Shell

目标：先搭出左侧导航 + 右侧工作区骨架。

范围：

1. 顶部栏保留当前全局指标。
2. 新增左侧一级/二级导航。
3. 右侧工作区先挂载旧内容或单个迁移页面。
4. 移动端或窄屏采用折叠导航，不强行保留完整侧栏。

完成标准：

- 桌面端不再依赖横向大 Tab 作为主要导航。
- 当前模块和二级页面在界面上有清楚位置。
- 旧后台核心功能仍可访问。

### Phase 2：API 中转作为首个迁移切片

目标：选择截图中最拥挤、业务边界又相对明确的 `API 中转` 先迁移。

原因：

1. 它已经天然拥有五个二级对象。
2. 当前横向结构最不适合它。
3. 可以同时防止 `ApiTransitAdminConsole.tsx` 继续长成第二个 10k 行后台组件。

完成标准：

- `API 中转` 的二级对象进入左侧导航或工作区内稳定二级结构。
- `站点池`、`清洗候选`、`原始报价`、`提交线索`、`检测记录` 的状态和请求边界清楚。
- 主 `AdminConsole` 不再直接持有 API 中转内部状态。

### Phase 3：审核队列迁移

目标：把后台最核心的待处理任务集中成“审核队列”。

范围：

1. 渠道提交。
2. 报价反馈。
3. 站点反馈。
4. 赞助审核。

完成标准：

- 待处理计数可在左侧导航中展示。
- 批量操作、单条详情和结果反馈保持可用。
- 未审核反馈和已审核反馈的语义不混淆。

### Phase 4：采集系统与数据维护迁移

目标：把运行状态类任务和数据维护类任务分开。

范围：

1. 采集待办、运行历史、健康状态进入“采集系统”。
2. 渠道源、标准商品、报价维护、官方价格进入“数据维护”。

完成标准：

- 采集排障不再需要在多个顶部 Tab 之间跳转。
- 数据维护页面保持表格优先、筛选稳定、编辑入口明确。
- 维护型页面不抢占审核队列的注意力。

### Phase 5：系统区收口与视觉统一

目标：处理低频系统能力，并统一后台视觉语言。

范围：

1. 安全。
2. 操作历史。
3. 站点公告。
4. 赞助配置。
5. 共享表格、toolbar、badge、empty/loading/error 状态。

完成标准：

- 低频设置不再挤压业务工作流。
- 后台按钮、表格、状态、筛选和消息条统一。
- 没有新出现的巨型组件或巨型 hook。

## 11. UI 验收标准

| 维度 | 验收标准 |
| --- | --- |
| 信息架构 | 一级分区不超过 6 个；二级入口按对象或队列命名 |
| 可扫描性 | 运营者 3 秒内能判断当前在哪个工作区、当前对象是什么、下一步能做什么 |
| 导航稳定性 | 切换页面不丢失全局登录状态和消息反馈；当前分区有明确 active 状态 |
| 内容密度 | 表格和列表保持后台工具密度，不做营销式大卡片布局 |
| 状态表达 | loading、empty、error、success、disabled 都有明确 UI 表达 |
| 响应式 | 宽屏使用固定左侧栏；窄屏折叠为抽屉或紧凑导航 |
| 可访问性 | 导航、筛选、行操作和详情区支持键盘焦点；不只靠颜色表达状态 |
| 回归控制 | 每个迁移切片都能独立验证，不要求整套后台一次性上线 |

## 12. 工程执行原则

1. 先行为等价迁移，再做视觉增强。
2. 先拆状态和请求边界，再抽视觉组件。
3. 不把 `AdminConsole` 变短但把复杂度搬进一个新巨型 hook。
4. 不在同一提交中同时迁移多个高风险工作流。
5. 每个切片都保留旧行为的验收清单。
6. 后台视觉改版与业务逻辑修复分开提交。
7. 新后台 feature 只能通过公开入口跨域引用，不直接互相 import 内部实现。

## 13. 风险与应对

| 风险 | 触发方式 | 应对 |
| --- | --- | --- |
| UI 改版和状态迁移互相干扰 | 同一阶段同时重排布局和重写请求 | 先 Admin Shell，再按工作流迁移 |
| 左侧导航变成另一种拥挤 | 一级分区过多或二级项无边界 | 一级按任务类型，二级按对象；低频项归入系统 |
| 主组件变短但复杂度未降 | 状态全部转移到一个 `useAdminConsole` | 每个工作流持有自己的 query/mutation 和局部状态 |
| 计数不一致 | 全局 summary 与局部页面各自请求 | 先定义计数来源和刷新策略，避免页面内随意重复 |
| 移动端不可用 | 左侧栏固定占宽 | 窄屏使用折叠导航，表格支持横向滚动或简化列 |
| 回归难定位 | 一次迁移多个旧 Tab | 每次只迁一个一级域或一个二级对象 |
