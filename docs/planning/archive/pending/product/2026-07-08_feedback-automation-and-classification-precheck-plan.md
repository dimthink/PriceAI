# PriceAI 用户反馈自动化与分类预审产品规划

> 文档类型：产品规划 / PRD Lite / 后续任务拆分基础
> 日期：2026-07-08
> 状态：待处理
> 适用范围：后台反馈工作台、分类规则、模型预审、自动复核、人工裁决、规则沉淀
> 核心目标：把“用户反馈 -> 人工排查 -> 修规则”的循环，升级为“用户反馈 -> 自动预审 -> 安全自动处理 / 人工确认 -> 规则沉淀”的闭环。

## 1. 背景

近期分类反馈暴露出一个典型问题：

```text
outlook/hotmail邮箱配合plus自助充值使用
```

它被归到了 `ChatGPT Plus`，但商品本体其实是 Outlook / Hotmail 邮箱，只是用途是配合 Plus 自助充值。这里的问题不是单个关键词，而是分类系统需要区分：

- 商品本体：用户实际买到什么。
- 使用场景：这个东西可以拿来做什么。
- 交付方式：成品号、充值、接码、邮箱、链接、CDK。
- 风险/售后：是否需要图片证据、是否能自动下架、是否只需重新采集。

如果继续每次都靠手工补规则，后续会在 Gemini、Grok、Claude、API/CDK、邮箱、接码、虚拟卡等分类里反复出现同类问题。

## 2. 产品目标

第一阶段不是让模型直接接管生产分类，而是让模型和规则一起服务后台运营：

1. 对分类问题自动给出建议分类、置信度和理由。
2. 对重复反馈按标题、来源、当前分类聚合，减少人工重复处理。
3. 对高置信度、低风险、可逆的纠错支持一键批量应用。
4. 对价格、库存、下架类反馈优先走采集/核验，不走分类模型。
5. 对售后、欺诈、渠道问题继续走风险预审和人工核验。
6. 把人工确认过的分类结论沉淀为 override 或正式规则，避免同类问题复发。

## 3. 反馈分流

| 反馈类型 | 当前问题 | 建议自动化路径 | 是否可自动改生产 |
| --- | --- | --- | --- |
| 分类错误 | 目前只标记“重建分类”，没有具体建议 | 分类预审模型 + 规则审计 + 重复反馈聚合 | 高置信度可一键应用，默认先人工确认 |
| 价格不准 | 可能只是后续采集已更新 | 自动比对当前价格，必要时创建重采任务 | 不直接改价，重采后自动判断 |
| 库存不准 | 可能是原站状态已变 | 自动重采 / 轻量打开原链接核验 | 可按核验结果下架单条报价 |
| 商品已下架 | 适合自动核验 | 现有自动核验继续增强 | 高置信度可下架报价 |
| 售后/发货 | 需要证据和上下文 | 风险模型预审 + 人工核验 | 不自动下架渠道 |
| 疑似虚假 | 需要高风险证据 | 风险模型预审 + 公开风险提示审核 | 不自动公开，需人工确认 |
| 渠道问题 | 影响店铺/来源信任 | 聚合来源级反馈，进入渠道观察池 | 不自动下架整个渠道 |
| 站点意见 | 多为产品建议或数据体验 | 转产品待办或忽略 | 不改报价数据 |

## 4. 分类预审模型

### 4.1 输入

分类预审每次拿到的信息应包括：

- 原始商品标题。
- 当前标准商品和平台。
- 用户反馈时所在的商品页。
- 用户备注。
- 来源名称、来源 ID、价格、库存状态。
- 现有规则 `classifyOffer()` 的结果。
- 候选标准商品列表，限制在同平台或相关辅助平台内。
- 同标题/相似标题的历史反馈数量。

### 4.2 输出

模型只输出结构化 JSON，不直接写库：

```json
{
  "decision": "suggest_reclassify",
  "currentProductId": "chatgpt-plus",
  "predictedProductId": "outlook-account",
  "confidence": 0.92,
  "commodityType": "email",
  "usageContext": "for_chatgpt_plus_recharge",
  "reason": "标题核心是 outlook/hotmail 邮箱，plus 是使用场景，不是交付的 Plus 订阅。",
  "matchedSignals": ["outlook", "hotmail", "邮箱", "配合plus使用"],
  "negativeSignals": ["没有成品号", "没有月卡", "没有会员", "没有直充"],
  "safeToAutoApply": true
}
```

### 4.3 决策类型

| 决策 | 含义 | 后台动作 |
| --- | --- | --- |
| `suggest_reclassify` | 模型认为当前分类错了 | 展示建议分类和理由，可一键应用 |
| `keep_current` | 模型认为当前分类没错 | 建议标记忽略，并显示解释 |
| `needs_rule` | 模型能判断错因，但不建议直接改库 | 进入规则待办，生成测试样例 |
| `manual_review` | 证据不足或候选冲突 | 保持人工处理 |
| `not_category_issue` | 其实是价格、库存、售后等问题 | 转到对应反馈队列 |

## 5. 自动应用安全线

第一版建议只做“半自动”，不要后台静默全自动改库。

允许一键批量应用的条件：

1. 反馈类型是分类错误。
2. 模型置信度不低于 `0.85`。
3. 目标标准商品存在且启用。
4. 当前报价没有高风险/售后争议。
5. 同一归一化标题有重复反馈，或规则结果与模型建议一致。
6. 修改只影响 `raw_offers.canonical_product_id` 和 `category_slug`，可通过迁移或审计日志回滚。
7. 应用后必须标记公开快照 dirty。

不允许自动应用的情况：

- 要下架渠道、隐藏店铺、公开风险提示。
- 模型建议跨到灰色或隐藏商品，但没有明确商品本体信号。
- 标题同时像多个主商品，例如 Plus + Team + Pro + API 混在一起。
- 价格异常、售后争议、诈骗反馈混合在同一条里。

## 6. 规则沉淀机制

模型预审不应该变成“永久在线猜分类”。推荐两层沉淀：

### 6.1 临时 override

新增 `classification_overrides`：

- 归一化标题 hash。
- 可选来源范围。
- 目标商品 ID。
- 证据数量。
- 来源：模型建议 / 人工确认 / 用户反馈聚合。
- 过期时间或复审时间。

适合处理高频重复标题，立即止血。

### 6.2 正式规则和测试

当同类 override 多次出现，转成：

- `src/lib/catalog.ts` 的正式规则。
- `scripts/test-catalog.mjs` 的回归样例。
- 必要时增加 Supabase migration 修历史数据。

这条路径适合长期稳定的商品表达，例如“邮箱配合 Plus 使用”。

## 7. 后台体验

反馈工作台建议增加一个“模型预审”区域：

- 当前分类。
- 模型建议分类。
- 置信度。
- 商品本体判断。
- 使用场景判断。
- 命中信号 / 排除信号。
- 同类反馈数量。
- 操作按钮：
  - 应用到当前报价。
  - 应用到同标题报价。
  - 加入规则待办。
  - 忽略反馈。
  - 转人工。

重复反馈应自动聚合，比如这次 `outlook/hotmail邮箱配合plus自助充值使用` 5 条反馈，可以在后台显示成一个问题组。

## 8. 分阶段路线

### P0：分类预审可见化

- 为 `wrong_category` 增加 `classificationPrecheck` 结果结构。
- 后台支持单条运行预审。
- 后台展示建议分类、理由和置信度。
- 不自动写库。

### P1：一键应用与重复反馈聚合

- 同标题/同来源/同当前分类聚合。
- 支持应用到当前报价或同标题报价。
- 写入审计日志。
- 标记公开快照 dirty。

### P2：override 沉淀

- 建 `classification_overrides`。
- 已确认 override 在新采集入库时优先命中。
- 后台支持关闭、过期、复审 override。

### P3：规则反哺

- 定期列出高频 override。
- 自动生成测试样例建议。
- 人工确认后合并到正式规则。

### P4：扩展到其他反馈

- 价格/库存/下架走自动核验和重采。
- 售后/欺诈/渠道问题走风险预审和来源信任。
- 站点意见进入产品待办。

## 9. 本次短期修复如何接入

本次规则修复可以作为分类预审的第一条基准样例：

| 标题 | 当前误分 | 正确分类 | 规则原则 |
| --- | --- | --- | --- |
| `outlook/hotmail邮箱配合plus自助充值使用` | ChatGPT Plus | Outlook / Hotmail 邮箱 | 邮箱是商品本体，Plus 是使用场景 |
| `Gmail---URL Chatgpt专用接码 可分裂` | 用户质疑不是 ChatGPT | Google / Gemini 接码 | 接码是商品本体，ChatGPT 是使用场景 |
| `GPT K12 team 只支持谷歌邮箱 gmail` | 用户认为是邮箱 | ChatGPT Team / Business | Team 是商品本体，Gmail 是账号载体 |

这些样例可以直接进入未来分类预审 prompt 的少量示例库。

## 10. 待确认问题

1. 分类预审第一版是否只在后台手动触发，还是反馈创建时自动触发。
2. 高置信度自动应用是否需要“同标题至少 2 条反馈”作为前置条件。
3. override 是否只按归一化标题生效，还是允许按来源店铺收窄。
4. 模型供应商是否复用现有风险预审配置，还是单独配置分类预审模型。
5. 用户端是否显示“分类已修复”，还是仅后台处理。
