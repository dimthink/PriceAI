# PriceAI 登录信任与全栈质量整改产品规划

> 文档版本：v0.2
> 更新时间：2026-07-15
> 当前阶段：主要本地整改已完成，待提交、migration 与生产确认
> 审计基线：`main@b84adda`，生产站点 `https://priceai.cc`
> 文档性质：跨模块产品整改总纲，不替代工程设计、数据库设计或运维手册
> 事实底稿：[PriceAI 全栈只读质量审计](../../../../priceai-full-stack-read-only-quality-audit-2026-07-15.md)

## 0. 核心结论

PriceAI 的轻量登录 V1 已经具备 Google OAuth、账户中心、反馈归属和检测任务归属，但还没有形成一条足够可靠的“登录前—登录中—登录后—数据退出”信任链。

本轮只读审计未发现已确认的 P0，也没有确认当前账户接口存在可直接利用的 IDOR；公开浏览仍然免登录，主要公开页面也没有因为登录接入而整体动态化。这说明当前产品方向是正确的，不需要推倒重来。

当前真正需要优先解决的，是已经进入用户核心旅程的 P1 问题：

1. 登录可能被错误返回地址、OAuth 取消循环、Session 刷新不完整和域名切换打断。
2. 检测报告默认公开，与既定的“默认私密、主动公开”产品约定冲突。
3. 检测任务的配额、超时、轮询和回收未形成可靠闭环，既影响用户，也可能放大第三方成本。
4. 高风险反馈在上传证据后才要求登录，OAuth 返回后草稿和已上传状态丢失，并可能留下孤儿对象。
5. 发布、采集、留存、跨调度租约和第三方调用预算存在断链，产品质量无法稳定地从本地验证传递到生产。

因此下一阶段不应继续扩张商家认领、积分、收藏、评论或更多身份能力，而应先完成一轮“信任基础设施整改”：

```text
公开浏览不受影响
-> 需要身份时再登录
-> 登录可以取消、失败和恢复
-> 返回原页面并恢复非敏感草稿
-> 用户资源默认私密且归属明确
-> 任务可超时、可回收、可解释
-> 用户可退出、导出和删除
-> 发布前后都有自动验证和成本护栏
```

## 1. 产品一句话

PriceAI 是一个面向 AI 订阅与中转 API 用户的公开 Web 比价和质量雷达；登录只用于需要身份归属的反馈、检测、报告和账户操作，本规划帮助这些用户在不牺牲公开浏览和缓存性能的前提下，稳定、安全地完成身份操作并掌控自己的数据。

## 2. 规划背景与场景问题

### 2.1 当前阶段判断

原《用户登录系统产品规划》已经完成 V1 产品方向验证：

- 搜索、比较、查看价格、阅读指南和浏览中转站继续免登录。
- 登录主要服务反馈追踪、检测任务、报告归属和后续账户能力。
- Google OAuth 作为轻登录入口，不把 PriceAI 变成强制注册网站。

本轮审计说明，当前问题不是“要不要登录”，而是“登录已经进入真实用户旅程后，产品是否能兑现连续、私密、可恢复、可退出的承诺”。

### 2.2 场景问题

| 场景问题 | 用户任务 | 当前阻碍 | 用户后果 | 产品机会 | 证据状态 |
| --- | --- | --- | --- | --- | --- |
| 用户从详情页发起登录 | 登录后回到原页面继续操作 | 返回地址校验不严，且部分深链被降级为列表页 | 被带离站点、丢失上下文或需要重新查找 | 建立严格站内回跳和完整深链恢复 | 已确认 |
| 用户取消 Google 登录 | 安全返回 PriceAI 并继续匿名浏览 | callback 未明确处理取消、缺少 code 和交换失败；登录页可能再次自动跳转 | 用户陷入重复登录链，没有清晰退出路径 | 建立可取消、可重试、可返回的登录结果页 | 已确认 |
| 用户长时间停留或 Session 过期 | 无感刷新，失败时明确重新登录 | Server Component 无法写 Cookie 时吞掉刷新异常，现有 middleware 不承担认证刷新 | 随机退出、状态不一致、受保护页面失效 | 只在认证相关路径建立 Session 刷新代理 | 已确认 |

| 用户提交高风险反馈 | 登录后继续原表单并提交证据 | 证据先上传，真正提交时才要求登录；OAuth 后 React 草稿丢失 | 重填、重复上传、放弃提交、产生孤儿对象 | 在敏感动作开始前登录，并恢复非敏感草稿 | 已确认 |

| 用户查看检测报告 | 默认只让自己查看，必要时主动分享 | 报告页仅凭 job ID 读取，没有落实私密状态 | 未授权的检测上下文可能被访问或转发 | 新报告默认私密，分享使用独立公开凭证 | 已确认 |
| 用户在多个标签页登录或退出 | 所有标签页及时同步状态 | 账户状态只在组件挂载时读取一次 | 一个标签页已退出，另一个仍显示旧状态 | 建立统一账户状态和跨标签页同步 | 已确认 |
| 管理员轮换密码 | 让旧密码立即失效 | 数据库密码不匹配后仍接受旧环境变量密码 | “修改密码”不能真正完成撤销 | 把环境密码改为独立、可审计的 break-glass 机制 | 已确认 |
| 运维发布新版本 | 先验证再进入生产 | 当前 Cloudflare workflow 直接发布后 smoke，preview/promote 链未接通 | 失败版本可能先承接生产流量 | 建立 upload、preview、smoke、promote 发布闭环 | 已确认 |
| 平台持续采集和检测 | 在预算内稳定运行 | Collector Runtime 同步失败，跨调度 lease、retention apply 和第三方预算未闭环 | 数据漂移、重复调用、存储和 Egress 增长 | 建立运行时一致性、租约、留存与预算控制 | 已确认或高风险疑点 |

## 3. 产品目标与边界

### 3.1 本阶段目标

1. 让登录成为可取消、可失败、可恢复的辅助流程，而不是阻断匿名浏览的漏斗。
2. 让反馈、检测任务和报告具备明确的用户归属、隐私状态和生命周期。
3. 让 Session 过期、弱网、重复提交、多标签页和异常回调都有稳定产品行为。
4. 在不读取根布局 Cookie、不让公共页面动态化的前提下，减少登录状态闪烁和重复认证请求。
5. 让高成本任务具备配额、超时、回收、限流、预算和降级机制。
6. 让发布、采集和数据库留存具备可验证的生产闭环。
7. 为账号退出、数据导出、账号删除和隐私说明补齐最小产品闭环。

### 3.2 本阶段不做什么

1. 不把搜索、筛选、比价、详情和指南改成登录后可见。
2. 不在根 Server Layout 或所有公共请求中读取用户 Cookie。
3. 不扩展收藏、评论、积分、消息中心、商家认领和复杂角色系统。
4. 不新增更多 OAuth Provider、手机号登录、密码登录或自建身份系统。
5. 不把本规划写成数据库字段、API 契约、migration 或 Cloudflare 规则清单。
6. 不趁整改进行全站视觉重构、路由重写或大型组件一次性重写。
7. 不在缺少控制台数据时承诺具体账单金额或节省比例。

### 3.3 产品形态判断

| 候选形态 | 是否采用 | 原因 |
| --- | --- | --- |
| 现有 PriceAI Web 站点 | 采用 | 登录、反馈、检测和账户能力已经嵌入现有浏览旅程 |
| 独立账户站点 | 不采用 | 会增加跨域 Session、回跳和用户理解成本 |
| App / 小程序 | 不采用 | 当前问题是 Web 信任链，不是缺少分发载体 |
| 浏览器插件 | 不采用 | 与本轮身份和报告治理无直接关系 |
| 后台质量与成本工作台 | 复用现有后台 | 运维视图应进入 `/admin` 的系统工作流，不建设独立产品 |

## 5. 用户任务

### 5.1 匿名用户任务

1. 不登录即可完成搜索、筛选、排序、详情阅读和外链跳转。
2. 看见登录入口，但不会被自动弹窗或强制跳转打断。
3. 执行需要身份的动作时，先知道为什么需要登录、登录后会回到哪里。
4. 取消登录后可回到原页面继续匿名浏览。

### 5.2 登录用户任务

1. 登录成功后回到原页面、原详情或原操作入口。
2. 恢复非敏感草稿；API Key 等敏感输入不持久化。
3. 在多个标签页中获得一致的登录、刷新和退出状态。
4. Session 过期时，优先无感刷新；无法刷新时保留上下文并要求重新登录。
5. 只读取自己的反馈、检测任务和私密报告。
6. 清楚知道“退出当前设备”和“退出全部设备”的区别。
7. 能查看隐私说明，并发起数据导出或账号删除。

### 5.3 管理与运维任务

1. 管理员修改密码后，旧密码必须真正失效。
2. 管理员可以主动退出，认证失败和高频尝试可被追踪和限制。
3. 发布者在生产流量切换前完成 preview smoke。
4. 运维者能确认主站版本、Collector Runtime、Supabase migration 和生产行为是否一致。
5. 维护者能看到检测、采集、R2、Supabase Egress 和第三方调用的预算与异常趋势。

## 6. 核心用户旅程

### 6.1 公开浏览与可选登录

```text
进入公开页面
-> 搜索 / 筛选 / 比较 / 查看详情
-> 需要账户能力时点击登录或执行受保护操作
-> 说明登录原因和返回位置
-> 登录成功后回到原上下文
-> 继续操作
```

关键原则：公共数据和页面渲染不依赖用户 Session；Header 可以在客户端异步显示账户状态，但不得让公共页面整体失去 CDN 缓存。

### 6.2 OAuth 登录、取消与失败

```text
发起登录
-> 记录经过校验的站内 return path
-> 跳转 Google OAuth
-> 成功：交换 Session，清理临时状态，返回原路径
-> 取消：回到登录结果页，允许返回原页面或重试
-> 失败：显示可理解错误，保留原路径，不自动循环跳转
```

| 步骤 | 用户动作 | 系统动作 | 用户获得什么 | 关键风险 |
| --- | --- | --- | --- | --- |
| 1 | 点击登录 | 记录规范化站内路径 | 明确登录后回哪里 | 开放重定向 |
| 2 | 在 Google 选择账号 | 使用 PKCE 完成授权 | 安全身份交换 | verifier Cookie 生命周期过长 |
| 3 | 取消或遇到错误 | 显示取消、缺参数、网络或交换失败状态 | 可返回、可重试 | 自动再次跳转造成循环 |
| 4 | 登录成功 | 建立 Session，并通知当前和其他标签页 | 账户状态一致 | Session 固定或多标签不同步 |

### 6.3 高风险反馈

```text
选择反馈类型
-> 系统判断是否需要登录与证据
-> 高风险类型先登录
-> 返回并恢复文本、选项和已确认的非敏感草稿
-> 逐项上传证据并显示每项状态
-> 提交反馈
-> 在账户中心追踪处理与补充材料
```

产品规则：

- 低风险数据纠错可继续匿名。
- 商家质量、稳定性、风险和需要公开归属的反馈必须登录。
- 登录判断发生在上传证据之前。
- 每个文件独立显示成功、失败、重试和移除状态。
- 未绑定正式反馈的上传对象必须有短期草稿状态和自动清理边界。

### 6.4 检测任务与报告

```text
登录
-> 填写检测参数
-> 确认隐私、额度和预计耗时
-> 原子领取配额并创建任务
-> 展示 queued / running / retrying / timed_out / failed / succeeded
-> 生成默认私密报告
-> 用户主动创建公开分享
-> 可撤销分享或删除记录
```

产品规则：

- 新检测报告默认私密。
- 公开分享与内部 job ID 分离，公开结果使用脱敏视图。
- API Key 不在浏览器持久化，不出现在公开报告、分析事件或日志中。
- 任务有明确过期时间、心跳和回收状态，不允许永久 queued/running。
- 轮询采用退避和终止条件，不让失败任务继续产生无意义请求。

### 6.5 Session 过期、退出与多标签页

```text
Session 即将或已经过期
-> 认证路径尝试刷新
-> 成功：继续当前操作并同步所有标签页
-> 失败：保存非敏感上下文，显示重新登录
-> 用户退出当前设备或全部设备
-> 所有标签页清理账户状态并回到可匿名浏览状态
```

## 8. 功能明细

### 8.1 认证可靠性

| 功能 | 支持能力 | 优先级 | 产品验收 |
| --- | --- | --- | --- |
| 站内回跳 | 只接受规范化 pathname、query 和 hash；拒绝外站、反斜杠网络路径和异常编码 | P1 | 所有登录入口只能返回 `priceai.cc` 站内路径 |
| 回调结果 | 区分成功、取消、缺 code、交换失败、网络失败和重复回调 | P1 | 任何失败都不会自动循环进入 OAuth |
| canonical 域名 | 认证前统一到 `priceai.cc`，`www` 只做稳定跳转 | P1 | 同一用户不会因主域切换出现登录态分裂 |
| Session 刷新 | 只在账户、认证和受保护操作路径刷新 Cookie | P1 | Session 过期可恢复，公共页面仍保持静态缓存 |
| OAuth 临时 Cookie | 显式设置 `Secure`、`HttpOnly`、`SameSite`、`Path` 和短有效期，完成或取消后清理 | P2 | verifier 不长期留存，也不能被前端脚本读取 |
| 统一账户状态 | Header、检测页和账户功能共享同一状态来源 | P2 | 单页面生命周期内不重复请求账户接口 |

## 11. 关键状态与异常状态

| 状态 | 用户应该看到什么 | 系统应该做什么 | 不允许出现 |
| --- | --- | --- | --- |
| 未登录浏览 | 完整公开内容和可选登录入口 | 保持公共缓存 | 强制跳登录 |
| 登录跳转中 | 明确正在前往 Google，可返回 | 锁定重复点击 | 多次创建 OAuth 流程 |
| OAuth 取消 | “已取消登录”，返回原页面或重试 | 清理临时 verifier / state | 自动再次跳转 |
| 回调失败 | 可理解错误和重试入口 | 保留安全 return path，记录脱敏错误 | 空白页或循环 |
| Session 刷新中 | 轻量 loading，不清空当前上下文 | 只在受保护路径刷新 | 全站动态化 |
| Session 失效 | 说明需要重新登录并保留非敏感草稿 | 清除旧状态、同步标签页 | 继续显示已登录但操作失败 |
| 无权访问他人资源 | 统一的不可枚举结果 | 不返回资源存在性或敏感字段 | 通过不同错误泄露资源 |
| 证据部分上传失败 | 每项结果、重试和移除 | 保留已成功项 | 全部显示失败并遗留对象 |
| 检测排队或运行 | 当前状态、等待建议和取消/离开说明 | 心跳、租约、退避轮询 | 90 次固定高频轮询后仍无解释 |
| 检测超时 | 明确终止、是否退还额度、何时可重试 | 回收配额和任务 | 永久占用并发名额 |
| 报告私密 | 私密徽标和分享入口 | 仅所有者可读 | 仅凭 job ID 公开读取 |
| 弱网重复提交 | “正在确认结果”，允许恢复 | 幂等识别 | 创建多条反馈或收费任务 |
| 管理员改密 | 明确新密码已生效 | 旧凭据撤销 | 旧环境密码继续登录 |

## 14. 分阶段整改路线图

### 14.1 0～2 天：信任止血包

目标：先关闭低成本、高确定性的认证、隐私和后台安全缺口，不改变公共产品结构。

1. 收紧 OAuth return path，只允许规范化站内路径。
2. callback 显式处理取消、缺参数、交换失败和重复回调，移除失败后的自动循环。
3. 统一认证 canonical origin，明确 `priceai.cc` 与 `www` 的跳转顺序。
4. 停止新检测报告默认公开；在完整权限模型落地前采用安全默认值。
5. 给检测状态请求补 origin/path 边界、timeout 和响应上限。
6. 管理员数据库密码存在时停止接受旧环境密码，并明确临时 break-glass 方案。
7. 将 Auth、权限、报告隐私和 callback 测试加入首批发布阻断清单。
8. 冻结商家认领、Token 福利和更多账户能力扩张，直到第一批验收通过。

### 14.2 1～2 周：完整用户旅程包

目标：让登录、反馈、检测和报告从入口到异常恢复形成闭环。

1. 迁移到 Next.js 16 认证路径专用 `proxy.ts`，完成 Session 刷新，并验证公共页面缓存不退化。
2. 建立统一账户状态、页面内请求去重和多标签页同步。
3. 高风险反馈改为登录前置，完成 OAuth 草稿恢复、逐项上传状态和孤儿对象 TTL。
4. 检测任务完成原子配额、任务租约、心跳、超时回收、重复提交保护和退避轮询。
5. 完成报告 owner 鉴权、私密/公开状态、脱敏分享和撤销。
6. 补账户、检测和后台局部 loading、error、empty、forbidden 状态。
7. 统一 Dialog、搜索标签、焦点样式和移动端表单交互。
8. 接通 Cloudflare upload、preview smoke、promote，并修复 Collector Runtime 同步。

### 14.3 1～2 月：数据权利、成本和长期质量包

目标：把一次性修复沉淀为长期可维护的产品基础。

1. 上线隐私说明、数据导出和账号删除请求流程。
2. 选择用户域隔离长期方案：用户 JWT + RLS，或强制 user scope 的 DAL/RPC，并补跨用户负向测试。
3. 建立检测、反馈证据、R2、Supabase Egress 和第三方调用的预算、告警、熔断与降级。
4. 为 API Transit、snapshot refresh 和 fallback 调度建立跨运行原子 lease。
5. 将 retention 从 dry-run 能力推进到单实例、有限批次、可观察的执行闭环。
6. 验证公共 API 的真实 Cloudflare edge hit ratio，优化重复请求、payload 和查询字段。
7. 完成长列表滚动恢复、搜索 debounce、首屏外图片 lazy loading 和关键 Web Vitals 复盘。
8. 按业务工作流拆分超大型 Client Component，但不做一次性全仓重构。
9. 建立月度登录质量、隐私访问拒绝、任务成本、发布成功率和采集漂移复盘。

## 16. 成本估算方法与控制台缺口

本规划只定义估算方法，不承诺当前费用。

### 16.1 Cloudflare

```text
Workers 月成本
= 基础套餐
+ 超额请求数 × 请求单价
+ 超额 CPU ms × CPU 单价
+ Logs
+ Durable Objects
+ R2
```

重点补充：

- Worker 月请求与按 route 拆分。
- CPU p50 / p95 / p99。
- HTML、RSC、`/api/explorer`、offers 和账户接口的 Cache hit / miss。
- Workers Logs events 和采样率。
- R2 feedback evidence 的 Storage、Class A、Class B 和孤儿对象。
- Durable Object queue 的请求、duration 和错误。

### 16.2 Supabase

```text
Database Egress
≈ 公共 API 请求数 × (1 - 实际边缘命中率) × 每次回源字节

Auth 成本
≈ 超出套餐额度的 MAU × 单价

数据库增长
≈ 每日新增行 × 平均行和索引字节 - 每日实际清理与复用字节
```

重点补充：

- 当前 plan、Spend Cap、MAU、Auth Egress、Database Egress 和 Cached Egress。
- Database size、Disk/WAL、连接峰值、Top queries 和平均返回字节。
- availability、detection、snapshot 和用户域表的行数、表/索引大小与 retention candidates。
- 生产 RLS policies 与 migration drift。
- Supabase Storage 是否存在遗留 bucket。

### 16.3 低、中、高量级估算

不直接假设用户量，统一按参数计算：

| 量级 | 月公开页面/API 请求 | 月登录用户 MAU | 月检测任务 | 估算用途 |
| --- | ---: | ---: | ---: | --- |
| 低 | `R_low` | `M_low` | `D_low` | 当前小规模运营 |
| 中 | `R_mid` | `M_mid` | `D_mid` | 登录和检测开始增长 |
| 高 | `R_high` | `M_high` | `D_high` | 需要套餐、预算和熔断决策 |

```text
月总成本
= Cloudflare(R, CPU, Logs, R2, DO)
+ Supabase(MAU, Egress, Disk, Storage, Compute)
+ 检测任务 D × 单任务平均第三方成本
+ 模型预审调用 × 单次 token / 图像成本
+ 采集源调用与代理成本
```

任何正式预算都必须带上实际缓存命中率、平均 payload、失败重试倍数和第三方单价。

## 17. 暂时不应投入的过度优化

1. 不做全站强制登录或把价格详情放入账户墙。
2. 不同时扩展手机号、邮箱密码、更多 OAuth、Passkey 和自建身份服务。
3. 不在信任基础未完成前开发收藏、评论、积分、签到和复杂消息中心。
4. 不因一次审计直接重写整个账户系统或迁离 Supabase Auth。
5. 不为解决少数状态闪烁把所有公共页面改成 SSR。
6. 不一次性拆完 `ProductOffersPanel`、`PriceExplorer` 和后台巨型组件。
7. 不在没有真实 Cache Analytics 前设计复杂多层缓存系统。
8. 不在没有 provider 账单前过早优化第三方模型单价。
9. 不为历史报告立即建设复杂内容审核和社交分享平台。
10. 不建设独立 App、账户站点或基础设施 SaaS。

## 18. 当前建议决策

建议直接采用以下产品口径：

1. 公开搜索、比较、详情和指南永久保持免登录优先。
2. 高风险反馈在上传证据前要求登录；低风险纠错保留匿名通道。
3. 新检测报告默认私密，用户主动生成公开分享，分享可撤销。
4. `priceai.cc` 为唯一认证 canonical origin，`www` 在进入 OAuth 前收敛。
5. API Key 不持久化，不进入公开报告、日志和分析事件。
6. 默认退出当前设备，另行提供“退出全部设备”的明确动作。
7. 普通用户认证与管理员认证继续完全分离。
8. 登录状态通过客户端共享状态呈现，不在根布局读取 Cookie。
9. 先完成 Auth、隐私和任务生命周期整改，再启动商家/站长权限扩张。
10. 所有高成本动作必须具备额度、timeout、回收、预算和降级。

仍需确认的产品决策：

1. 历史检测报告的迁移口径：保持现状、统一私密，还是按已有外链分批处理。
2. 账号删除第一版是自动自助，还是“用户提交请求 + 管理员审核”的过渡方案。
3. 数据导出第一版的交付形式和处理时限。
4. 检测超时后是否退还日额度，以及哪些失败状态不计费。
5. break-glass 管理员入口由 Cloudflare Access、独立 secret 还是人工轮换流程承接。

## 23. 最终残余整改的收益、缺点与风险

| 改动项 | 主要收益 / 优点 | 确定缺点 / 代价 | 影响范围 | 风险与回滚 | 验证方法 |
| --- | --- | --- | --- | --- | --- |
| 公开 API 缓存键规范化 | 无关 query 不再制造无限 MISS，减少 Worker CPU、Supabase 回源和 Egress | 每条缓存路由必须维护真实参数白名单 | explorer、health、全站报价、商品报价 | 漏掉业务参数会错误共享缓存；可补白名单并提升 namespace 版本 | Preview 对 `nonce=1/2` 连续请求应共享 HIT；有效筛选仍返回不同结果 |
| 账号删除独立 workflow | 单个 R2/Auth 删除失败不再阻断公开快照兜底 | 增加一条 workflow、独立告警和少量 Actions 调度 | GitHub Actions、账户删除、公开快照 | 配置错误只影响删除任务；可单独禁用 schedule，不影响 snapshot | YAML 解析；分别触发两个 workflow，确认失败域互不传播 |
| 报告分享错误语义分离 | 数据库故障返回 500，资源不存在才返回 404，便于重试与告警 | 客户端需接受明确的服务不可用状态 | 账户检测报告分享 API | 误判 Supabase error 会影响错误提示；可回滚单 route | mock DB error/empty row，分别断言 500/404 |
| 证据 URL 限制为 HTTP(S) | 阻止 `javascript:`、`data:`、`file:` 等危险链接进入账户和后台 | 用户不能提交自定义协议；历史非法值只显示无效文本 | 反馈补充、账户页、后台 | 过严规则可能拦截非标准证据平台；只扩展明确安全协议 | schema 与 UI 单测；手工输入危险 scheme |
| 管理员证据 `no-store` | 降低退出后台后从浏览器缓存恢复敏感截图的概率 | 每次查看都读取 R2，增加少量 Class B 请求和延迟 | 管理员证据查看、R2 | 重复预览成本略升；不得恢复浏览器持久缓存 | 响应头包含 `private, no-store`、`nosniff` 和 sandbox CSP |
| runtime lease 续租 | 长采集超过原固定租期时仍保持单实例，降低重复写入和第三方调用 | 增加周期性 Supabase RPC；网络故障需观察 heartbeat 日志 | API Transit、官方价格、runtime leases | 心跳参数漂移可能造成过早失租；可停 heartbeat 并临时提高固定租期 | SQL 隔离测试 claim/renew/release；长时 dry-run 观察 heartbeat |
| 上传持久配额 | 多 PoP、多 isolate 共享配额，避免内存限流绕过 | 每次上传增加一次数据库 RPC；数据库不可用时上传失败关闭 | 反馈图片、Workers、Supabase | RPC 热点或 migration 未落地会返回 5xx；可临时切 Cloudflare Rate Limiting | 并发调用 quota RPC，断言第 31 次 429 和 `Retry-After` |
| Supabase recovery baseline | 新环境可从固定 schema 恢复，不改写生产 migration 历史；CI 阻止漂移 | 新 migration 必须同步 schema/checksum/head；不等同于标准 `db reset` 纯历史重放 | 灾备、Preview、Supabase Integration、CI | 只改 hash 不实测会掩盖顺序问题；恢复仅可在空库审批执行 | `npm run check:recovery-baseline -- --docker` 在临时空库应用成功 |
