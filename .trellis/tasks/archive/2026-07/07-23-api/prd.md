# 修复 API 中转监测过期数据误导

## 背景

采集恢复后，部分站方公开 API 持续返回 `404` 或其他错误。数据库会保留最后一次成功的可用性样本，但列表的“更新时间”仍随失败检查前进，旧样本也继续以正常绿色稳定性展示，容易让用户误以为最近仍有监测数据。

## 目标

- 将站点采集状态、错误和最近检查时间透传到公开列表数据。
- 用统一的新鲜度规则区分正常、延迟、停止和从未监测；过期样本只作为历史证据展示。
- 站方公开监测过期但 PriceAI 自测仍新鲜时，明确切换到 `PriceAI 实测`，不混淆来源。
- 停止的监测证据不参与稳定性与综合排序。
- 站方接口中断时，充值倍率、模型倍率和综合倍率降级为历史证据；过期价格退出价格排序与综合推荐。
- 将列表“更新时间”明确为价格更新；采集失败时单独展示最近检查和同步失败。

## 验收

- API transit 相关测试、lint、typecheck 和 build 通过。
- 本地 `/api-transit` 可清楚看到“暂无近期监测/监测延迟”、历史样本和来源标签。
- `404/410` 且监测已明显断档，或最后监测超过 24 小时时，不再展示正常绿色稳定性条。
- 新鲜 `PriceAI 实测` 可作为明确标注的列表回退证据；旧公开监测不参与排序。
- 过期综合倍率显示“暂无近期价格”和历史值，延迟倍率显示“价格待更新”；两者不再伪装成刚确认的当前价格。
- 不修改生产数据库 schema，不推送或部署生产。
