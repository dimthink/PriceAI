# 商品详情库存与更新时间快捷筛选

## 背景

热门标准商品的报价数量较多，用户需要优先找到库存较多、最近成功确认过的报价，而不是逐行查看库存与更新时间。

## 目标

- 在商品详情筛选栏提供 `库存 >=50` 与 `1小时内更新` 两个常驻快捷条件。
- 在完整筛选中提供库存与更新时间固定档位。
- 快捷条件、完整筛选、URL、分页、缓存与数据库查询共用同一套状态和口径。

## 产品规则

- 快捷条件排在商品属性标签之前，移动端首屏可见。
- 两个条件可独立开启，同时开启时按 AND 组合。
- 库存档位：不限、>=10、>=50、>=100；库存未知不满足任何库存下限。
- 更新时间档位：不限、30分钟内、1小时内、3小时内、24小时内。
- 更新时间按公开报价口径 `verifiedAt -> lastSeenAt -> capturedAt -> sourceUpdatedAt` 对应的最近成功确认时间判断。
- 选中快捷条件后，完整筛选中的对应档位同步选中；反向修改也同步快捷条件。
- URL 使用稳定查询参数，刷新、分享和返回页面后保留筛选条件。
- 筛选后保持现有有货优先、特殊交付后置、低价优先排序。

## 边界

- 不新增“库存充足”“可抢购”等可信度或库存状态标签。
- 不把库存未知当成 0，也不把采集失败时间当成成功更新时间。
- 不修改采集频率、库存采集逻辑或外层商品最低价规则。
- 不在客户端全量读取报价后过滤。

## 验收

- 桌面端和 390px 移动端均能看到并操作快捷条件，文字不溢出。
- 完整筛选可选择全部库存和更新时间档位，并显示活动筛选摘要。
- URL、API cache key、客户端 memory/session cache key 包含新条件。
- 默认与带筛选的数据库分页 RPC 均保持有界查询和原有排序。
- fallback 数据路径与数据库路径产生一致结果。
- 无效查询参数回退为不限，不向前台暴露数据库错误。
- lint、相关测试、typecheck/build 通过。
